# llm-d v0.7: From Feature Introduction to Production Hardening

In v0.6, llm-d demonstrated what intelligent orchestration could achieve: OTel integration, prefill/decode disaggregation, and multi-accelerator support validated the architecture. Version 0.7 shifts focus from proving capabilities to making them deployable.

Changes touch every layer—deployment tooling, hardware support, documentation, continuous integration. The unifying theme is operational maturity: remove friction, broaden hardware reach, give operators the documentation and CI coverage needed to trust the system in production.

This is the release where llm-d stops being a collection of promising features and starts looking like production infrastructure.

<!-- truncate -->

## Documentation Built from Scratch

To ensure teams can onboard without tribal knowledge, v0.7 introduces **124 changed markdown files with over 10,000 lines of new content**. This is not incremental improvement—it's a complete documentation rebuild.

**Architecture reference**: Full coverage of EPP design (scheduling, flow control, request handling), KV management subsystems, disaggregation operations, autoscaling mechanics, and the latency predictor. New API reference sections document HTTP headers, CRD schemas, and configuration patterns.

**Visual topology**: Fifteen new SVG diagrams make system architecture legible at a glance. Each major subsystem—KV cache indexing, prefill/decode coordination, multi-tier offloading—has visual representations showing component interaction and data flow.

**Operational tooling**: Interactive tuning scripts for flow control let operators explore parameter spaces. A new health check script validates deployments end-to-end, catching configuration errors before they surface in production.

The guide taxonomy was restructured for clarity: "inference-scheduling" became "optimized-baseline" (reflecting what it delivers), benchmarks moved to a helpers directory, and 164 new guide files fill gaps that previously required reading source code.

## Deployment Simplification

**Standalone mode as default**: Previously, getting llm-d running meant deploying the full Gateway API Inference Extension (GAIE) stack before sending your first request. While that power remains available (GAIE v1.5.0, Istio 1.29.2, AgentGateway v1.1.0), standalone mode now gets you from repository clone to serving requests in minutes using a generic Envoy proxy. Layer on gateway features as deployment requirements evolve.

**Kustomize-native installation**: Every guide migrated from Helm to Kustomize bases with per-platform overlays. For teams managing Kubernetes with GitOps tools like ArgoCD or Flux, this eliminates a tooling boundary—llm-d configurations become first-class citizens in existing deployment pipelines. Platform-specific patches (node selectors, storage classes, NCCL tuning) compose naturally without maintaining separate charts.

## Hardware: Blackwell to Rebellions

**NVIDIA Blackwell support**: CUDA 13.0.2 (breaking change, requires driver 580+) unlocks Blackwell-generation hardware. Dedicated `llm-d-cuda-gb200` images ship with GB200-specific DeepEP wheels, NVSHMEM, and native sm100 targets. Core inference stack updates: vLLM 0.19.1 (neuralmagic fork + NaN fixes), LMCache v0.4.4, FlashInfer v0.6.6.

**Expanded accelerator coverage**: Rebellions joins as supported vendor. AMD ROCm gains deployment guide. TPU support spans v6e and v7 with RunAI model streamer. Intel XPU adds Tensor Parallelism (TP=2) for Qwen3-32B, validated in nightly tests. HPU and CPU images updated.

For mixed-accelerator fleets—NVIDIA + AMD + Intel + Google TPU + Rebellions—v0.7 is the first release where every platform ships with tested images, validated guides, and production documentation.

## Multi-Tenant Infrastructure (Experimental)

Two features address production challenges beyond single-team deployments.

**Flow Control** moves request queuing to the Router for intelligent admission control. Requests are classified by FlowKey (Fairness ID + Priority) and dispatched via priority → fairness → ordering. This prevents noisy neighbors, enables "no-regret scheduling" during saturation, and integrates with InferenceObjective for SLO enforcement. Ships with FCFS-compatible defaults for seamless transition.

**Batch Gateway** provides OpenAI-compatible `/v1/batches` and `/v1/files` APIs for jobs up to 50,000 requests. Separates batch from interactive traffic while coexisting on shared infrastructure. Pluggable storage (PostgreSQL/Redis/S3), multi-tenant isolation, three-component architecture (API Server, Batch Processor, Garbage Collector).

Both features ship experimental, reflecting operational complexity and need for real-world validation before production graduation.

## KV Cache: Routing to Tiering

**Precise prefix cache-aware routing**: Speculative indexing now default, removing hashSeed workarounds. Router tracks cache state real-time, scores endpoints by prefix overlap, routes to pods with relevant blocks in GPU memory. Validated on CKS, OpenShift, Intel XPU (TP=2 Qwen3-32B).

**Tiered storage for long contexts**: Multi-tier offloading (GPU HBM → CPU DRAM → persistent storage). AWS EFS backend guide for cloud deployments. Integrates with vLLM's OffloadingConnector—transparent to model servers, managed by router.

**Architecture documentation**: KV Indexer (distributed block index), KV Offloader (tier coordination), routing plugin integration. Diagrams show component interaction and data flow for operators debugging performance.

## Prefill/Decode Disaggregation

Platform coverage expands: **Google TPU v7** (Qwen 3.5), **Oracle Cloud Infrastructure**, comprehensive three-part architecture series (KV cache transfer, NIXL library, routing sidecar coordination). Guide rewritten with Kustomize, per-platform overlays for NVIDIA (vLLM/SGLang), TPU, OCI.

---

## Autoscaling

Five-part documentation series: fundamentals, HPA/KEDA design choices, WVA deep-dive, native HPA patterns, guide alignment. Updated to reference EPP metrics (not deprecated InferenceGateway path) for clearer autoscaling signals. Clarifies when WVA's variant-aware logic provides value over simpler approaches.

---

## CI That Tests What You Deploy

Complete CI rewrite: monolithic 550-600 line workflows replaced with focused per-guide nightly matrices (optimized baseline, precise prefix cache, predicted latency, tiered offload) parameterized across CKS, GKE, OpenShift.

Predicted latency validation exemplifies rigor: scrapes EPP `/metrics` to verify predictor served real predictions, not heuristics. This asserts the optimization path you configured actually runs.

Image verification workflow, `/test-nightly` slash command with globs, badge matrix generator, PR status comments. 63 workflow files changed, net **2,785 fewer lines of YAML**—consolidation while covering more deployment surface.

## What This Means

If you evaluated llm-d before and concluded it wasn't production-ready, v0.7 merits a second look:

- **Deployment path simplified**: Standalone mode + Kustomize-native installation
- **Hardware coverage**: Blackwell, AMD ROCm, Intel XPU, Google TPU, Rebellions, CPU—tested images and guides for each
- **Documentation**: Onboard without tribal knowledge or source archaeology
- **CI confidence**: Nightly tests validate guides work on claimed platforms

23 new contributors this release (3.5× PR volume over v0.6). The project is accelerating. The v0.7 investment in foundations—documentation, testing, installation—compounds as the project scales.

## Looking Ahead

[v0.8 milestone](https://github.com/llm-d/llm-d/issues/1520) targets mid-June with five themes:

**Production graduation**: Flow Control and Batch Gateway move experimental → production. Multi-modal serving (LLaVA, Qwen-VL) gains guides and E2E tests.

**RL infrastructure**: Python scheduler integration (gym-style RL → routing decisions), non-Kubernetes mode (Slurm/research labs via file-based discovery), GPU time-slicing (multiple RL rollout workers share accelerators).

**Advanced disaggregation**: NIXL TTL + request draining, new connectors (Mooncake/MoRI), GB200 NVL72 guides, object storage backends for multi-tier offload.

**CI/operational rigor**: TPU + AMD nightly tests, automated lm-eval per guide, documented release process, streamlined monitoring dashboards.

**Deployment flexibility**: Multi-model guides (LoRA scheduling), rollout patterns (blue/green, zero-downtime), consolidated gateway prerequisites.
