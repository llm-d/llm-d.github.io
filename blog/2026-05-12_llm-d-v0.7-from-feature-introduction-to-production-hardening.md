# llm-d v0.7: From Feature Introduction to Production Hardening

In our previous release (v0.6), llm-d demonstrated what intelligent orchestration could achieve for distributed LLM inference: OTel integration for observability, prefill/decode disaggregation for throughput optimization, and initial multi-accelerator images proving hardware flexibility. That work established llm-d's technical foundation and validated core architectural decisions.

Version 0.7 shifts focus from proving capabilities to making them deployable. Changes touch every layer of the stack—from deployment tooling and hardware support to documentation and continuous integration. The unifying theme is operational maturity: remove deployment friction, broaden hardware reach, and give operators the documentation and CI coverage needed to trust the system in production environments.

This is the release where llm-d stops being a collection of promising features and starts looking like production infrastructure.

## Deployment Simplification: Standalone Mode as Default

One of the most impactful changes for teams evaluating llm-d is the introduction of standalone mode as the default deployment pattern. Previously, getting llm-d running meant deploying the full Gateway API Inference Extension (GAIE) stack, configuring InferencePools, and understanding the gateway layer before sending your first request. While that power remains available—GAIE has been updated to v1.5.0, Istio to 1.29.2, and AgentGateway to v1.1.0—it is no longer the first thing you encounter.

Standalone mode uses a generic proxy (Envoy) to route requests directly to the llm-d Router, with all the intelligent routing, prefix-cache awareness, and autoscaling capabilities intact. The difference is that you can go from repository clone to serving requests in minutes, then layer on gateway features as your deployment matures and requirements grow more complex.

This change responds directly to feedback from platform teams who found the initial setup had too many moving parts. The complexity remains when you need it, but the on-ramp is now straightforward.

## Kustomize-Native Installation: One Pattern, Every Guide

The entire installation surface has migrated from Helm and Helmfile to Kustomize. Every deployment guide—optimized baseline, prefix-cache-aware routing, prefill/decode disaggregation, workload-variant autoscaling, Intel XPU deployment—now uses Kustomize bases with per-platform overlays for GKE, OpenShift, and other Kubernetes distributions.

The previous llm-d-infra and llm-d-modelservice Helm charts are deprecated, and the monolithic Helmfile CI workflow has been replaced with `ci-kustomize-dry-run.yaml`. Every guide has been rewritten to match this new structure.

The technical reasoning is pragmatic: Kustomize overlays compose more naturally than Helm value files when patching platform-specific details like node selectors, storage classes, or NCCL tuner configurations without maintaining separate charts. For teams already managing Kubernetes resources with GitOps tools like ArgoCD or Flux, this eliminates a tooling boundary and makes llm-d configurations first-class citizens in existing deployment pipelines.

## Hardware Breadth: From Blackwell to Rebellions

Platform support in v0.7 spans the full range of production LLM hardware. The headline change is the jump from CUDA 12.9.1 to 13.0.2—a breaking change requiring NVIDIA driver 580 or later—which unlocks first-class support for Blackwell-generation hardware. A dedicated `llm-d-cuda-gb200` Docker image ships with GB200-specific DeepEP wheels, NVSHMEM support, and native sm100 targets in `TORCH_CUDA_ARCH_LIST`.

Alongside the CUDA bump, the core inference stack has been updated: vLLM moves from 0.17.1 to 0.19.1 (now tracking the neuralmagic fork with targeted NaN-fix cherry-picks), LMCache to v0.4.4, and FlashInfer to v0.6.6. Build infrastructure improvements include sccache in the runtime stage and `MAX_JOBS=3` as default to prevent out-of-memory failures during source builds.

Beyond Blackwell, this release adds Rebellions as a supported accelerator vendor, ships an AMD ROCm deployment guide, and updates HPU, XPU, and CPU images. TPU support now spans v6e and v7 with consistent configurations across generations, including a new RunAI model streamer integration. Intel XPU support gains Tensor Parallelism (TP=2) for Qwen3-32B, validated through nightly prefix-cache tests.

If you are evaluating llm-d against a mixed-accelerator fleet—combining NVIDIA GPUs with AMD MI-series, Intel XPUs, Google TPUs, or Rebellions ATOM chips—v0.7 is the first release where every supported platform ships with tested images, validated guides, and production-grade documentation.

## Workload-Aware Routing: Multi-Tenancy and Batch Processing

Two new experimental features address distinct production challenges that emerge when moving beyond single-team, single-workload deployments.

### Flow Control: Intelligent Multi-Tenant Queuing

Traditional load balancing treats all requests identically and commits them immediately to backend queues. This breaks down for LLM inference where resource consumption varies wildly per request—a 32K-token prefill followed by minimal decode consumes radically different resources than a balanced workload—and where multi-tenant environments require fairness guarantees.

Flow Control moves request queuing to the llm-d Router, enabling intelligent admission control before requests reach model servers. Incoming requests are classified by a FlowKey combining Fairness ID and Priority. The Endpoint Picker (EPP) maintains separate in-memory queues for each flow and dispatches based on three criteria:

1. **Priority**: Service highest-priority bands first
2. **Fairness**: Cycle through tenants within a band  
3. **Ordering**: Maintain request sequence within a flow

This architecture prevents noisy neighbors from starving other tenants, enables "no-regret scheduling" that holds requests during saturation rather than committing them to stuck local queues, and integrates with InferenceObjective resources for dynamic SLO enforcement. While backpressure protects hardware from overload, Flow Control policies determine exactly how delayed traffic is ordered and distributed among users.

The feature ships with defaults explicitly designed to mimic legacy first-come-first-served behavior, ensuring a seamless transition for existing workloads while enabling operators to gradually introduce priority and fairness policies as requirements evolve.

### Batch Gateway: OpenAI-Compatible Batch Inference

Production LLM deployments rarely serve only interactive traffic. Batch workloads—large-scale embedding generation, synthetic data creation, offline document processing—represent significant compute demand but have fundamentally different latency tolerance and scheduling requirements than chat or completion endpoints.

The new Batch Gateway provides an OpenAI-compatible `/v1/batches` and `/v1/files` API for submitting and tracking batch jobs containing up to 50,000 requests. The architecture separates batch from interactive traffic while enabling both to coexist on shared infrastructure through integration with flow control backpressure mechanisms.

Three components handle the batch lifecycle:

1. **API Server**: REST endpoints for job submission, management, and file handling
2. **Batch Processor**: Pulls jobs from a priority queue, builds per-model execution plans, dispatches to llm-d Router, writes results to output files
3. **Garbage Collector**: Cleans up expired jobs and files based on configurable retention policies

Storage is pluggable across layers: PostgreSQL or Redis/Valkey for metadata, Redis/Valkey for queue and event streams, and S3 or filesystem for input and output files. Multi-tenant isolation is enforced through pluggable authentication and authorization with per-tenant data boundaries.

Both Flow Control and Batch Gateway ship as experimental features, reflecting their operational complexity and the need for real-world validation before stabilization. They represent llm-d's evolution from single-workload optimization to multi-tenant platform infrastructure.

## KV Cache Intelligence: From Routing to Tiering

Version 0.7 significantly expands KV cache management capabilities across three dimensions: more intelligent routing, multi-tier storage, and comprehensive architectural documentation.

### Precise Prefix Cache-Aware Routing

The precise-prefix-cache-aware guide now enables speculative indexing by default, removing hashSeed and PYTHONHASHSEED workarounds that previously limited cache effectiveness. The router tracks cache state in real-time across the fleet, scoring candidate endpoints based on prefix overlap and routing to pods most likely to have relevant blocks already resident in GPU memory.

This routing intelligence has been validated across platforms—passing nightly tests on both Container Kubernetes Service and OpenShift—and now extends to Intel XPU deployments where TP=2 configurations for Qwen3-32B are fully supported and tested.

### Tiered Storage for Larger Contexts

The tiered-prefix-cache guide demonstrates multi-tier KV cache offloading where blocks evict from GPU HBM to CPU DRAM to persistent storage (local NVMe or network-attached storage like AWS EFS). This enables handling longer contexts than would fit in GPU memory alone, trading storage I/O latency for the ability to preserve and reuse expensive prefill computation.

A new AWS EFS backend guide provides a reference implementation for cloud deployments needing shared storage across nodes. The tiered storage architecture integrates with vLLM's OffloadingConnector, making the optimization transparent to model servers while the router manages tier placement and retrieval.

### Architectural Documentation

New documentation sections explain the KV cache management subsystem in depth: how the KV Indexer maintains a distributed index of block locations across pods, how the KV Offloader coordinates tier transitions, and how routing plugins consume this state to make cache-aware placement decisions. Fifteen new architecture diagrams make system topology comprehensible at a glance, providing the conceptual grounding operators need to reason about performance and troubleshoot issues.

## Prefill/Decode Disaggregation: Expanded Platform Coverage

Prefill/decode disaggregation—where prefill computation (token-in, compute-bound) runs on separate pods from decode (token-out, memory-bandwidth-bound)—enables higher throughput by allowing independent scaling of each phase and eliminating prefill interference with decode latency.

Version 0.7 extends P/D support to Google TPU v7 with a complete guide for Qwen 3.5, adds an Oracle Cloud Infrastructure deployment path, and ships a comprehensive three-part disaggregation architecture series explaining how KV cache transfer works, what NIXL (NVIDIA Inference Xfer Library) provides, and how the routing sidecar coordinates transfer between prefill and decode instances.

The guide has been rewritten with Kustomize and now includes per-platform overlays, making it straightforward to deploy the same disaggregated topology across NVIDIA GPUs (vLLM and SGLang), Google TPUs, or Oracle Cloud Infrastructure with platform-specific tuning isolated in overlay patches.

## Autoscaling: From Configuration to Understanding

Autoscaling documentation received a comprehensive overhaul in the form of a five-part series covering autoscaling fundamentals, design trade-offs, and implementation patterns:

1. **Introduction**: Overview of autoscaling approaches for LLM inference
2. **HPA/KEDA design choices**: When to use native Horizontal Pod Autoscaler versus custom controllers
3. **Workload Variant Autoscaler (WVA)**: Deep-dive on llm-d's variant-aware autoscaler
4. **HPA with InferencePool Gateway**: Native HPA integration patterns
5. **Guide alignment**: Updated WVA guide matching optimized-baseline and Kustomize patterns

Native HPA guides were updated to reference EPP metrics instead of the deprecated InferenceGateway metrics path, providing a clearer signal for autoscaling decisions based on actual routing layer state rather than proxy-level approximations.

The documentation clarifies when WVA's variant-aware logic provides value over simpler approaches, helping operators choose the right autoscaling pattern for their workload characteristics and operational constraints.

## Documentation and Developer Experience

The documentation overhaul represents the largest single category by volume: 124 markdown files changed with 10,319 lines added. The docs/ directory gained 72 new files, including complete architecture references covering EPP design (scheduling, flow control, request handling), disaggregation operations, KV management, autoscaling mechanics, and the latency predictor.

A new API reference section documents EPP HTTP headers, EndpointPickerConfig, InferenceModelRewrite, InferenceObjective, and InferencePool resources. The guides directory added 164 files, and the entire guide taxonomy was restructured: what was "inference-scheduling" is now "optimized-baseline" (reflecting what it actually delivers), benchmarks moved to a helpers directory with comprehensive references, and a new smoke-test health check script provides turnkey deployment validation.

New guides for batch gateway, flow control (with an interactive tuning wizard script), and predicted latency routing fill gaps that previously required reading source code. Fifteen new SVG architecture diagrams make complex topologies legible, providing visual entry points for understanding system design.

## CI That Tests What You Actually Deploy

The continuous integration story in v0.7 is a complete rewrite. The previous approach—monolithic accelerator test workflows of 550-600 lines each—has been replaced with focused, per-guide nightly test matrices. New nightly jobs cover optimized baseline, precise prefix cache, predicted latency, and tiered prefix cache with CPU offloading, each parameterized across CKS, GKE, and OpenShift.

The predicted latency validation workflow exemplifies the new rigor: the test script scrapes EPP `/metrics` endpoints to verify the predictor served real predictions rather than falling back to heuristics. This is not smoke testing; it asserts that the optimization path you configured is actually running.

Additional CI improvements include an image verification workflow, a `/test-nightly` slash command with glob patterns enabling fork contributors to trigger targeted nightly runs, and a badge matrix generator keeping the README status table synchronized with actual test results. Nightly results now post status comments directly on pull requests, surfacing failures immediately rather than requiring manual log inspection.

Across 63 changed workflow files, the net delta is actually 2,785 fewer lines of YAML—consolidation rather than expansion—while covering substantially more of the deployment surface.

## What This Means for Your Evaluation

If you evaluated llm-d previously and concluded it was not ready for production use, v0.7 merits a second look. The deployment path is simpler through standalone mode and Kustomize-native installation. Hardware coverage spans Blackwell GPUs, AMD ROCm, Intel XPUs, Google TPUs, Rebellions ATOM, and CPU deployments. Documentation is comprehensive enough to onboard a team without requiring tribal knowledge or source code archaeology. The CI matrix covers enough deployment surface to provide confidence that guides actually work on the platforms they claim to support.

This release represents a 3.5× increase in pull request volume over v0.6, driven by 23 new contributors joining the existing community. The project is accelerating, and the v0.7 investment in foundations—documentation, testing, installation simplification—is the kind of work that compounds in value as the project scales.

## Looking Ahead

Version 0.7 establishes the operational foundation for increasingly sophisticated production features. Areas of active development for upcoming releases include:

**Cross-accelerator disaggregation**: Enabling prefill to run on one accelerator type (e.g., H100 for compute) while decode runs on another (e.g., MI325X for memory bandwidth), maximizing each phase's match to hardware strengths.

**Larger-scale P/D validation**: Testing disaggregation at 120B+ parameter models with longer contexts where the prefill-decode separation provides the largest efficiency gains.

**Flow Control stabilization**: Moving multi-tenant queuing from experimental to production-ready based on real-world validation and operational feedback.

**Batch Gateway maturity**: Production hardening of the OpenAI-compatible batch API with enterprise storage backends and at-scale validation.

**Non-Kubernetes deployment modes**: Enabling llm-d Router to run outside Kubernetes for bare-metal, HPC scheduler integration, and Ray-based workflows where Kubernetes overhead is undesirable.
